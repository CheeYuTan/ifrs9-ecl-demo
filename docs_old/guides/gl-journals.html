GL Journal Generation

Automated double-entry journal entries for ECL provisioning, with posting, reversal, and trial balance.

Generating Journals

Journal entries are generated from the ECL calculation results. For each product type and stage, the system creates balanced debit/credit entries:

POST /api/gl/generate/{project_id}
{
  "user": "finance.controller@bank.com"
}

The generated journal includes:

  • ECL provision entries — debit Impairment Charge (4100), credit Stage Provision (1210/1220/1230)
  • Overlay entries — debit Overlay Expense (4130), credit Overlay Provision (1240)
  • Each line tagged with product type and stage for drill-down analysis

Journal Lifecycle

Draft ──▶ Posted ──▶ Reversed (optional) │ └──▶ Reversal Journal (new draft)

Posting a Journal

POST /api/gl/journal/{journal_id}/post
{
  "user": "finance.controller@bank.com"
}

Reversing a Journal

POST /api/gl/journal/{journal_id}/reverse
{
  "user": "finance.controller@bank.com",
  "reason": "Correction required after data restatement"
}

Reversal creates a new journal with opposite debit/credit entries and links back to the original.

Trial Balance

View the trial balance for a project to verify that all entries are balanced:

GET /api/gl/trial-balance/{project_id}

Returns the total debits and credits per account, with a net balance. Total debits must equal total credits.

GL Reconciliation

The Data Processing step includes a GL reconciliation check that compares:

  • ECL amounts from the calculation engine
  • Journal entry totals from the GL
  • Differences flagged for investigation
GET /api/data/gl-reconciliation
Double-Entry Guarantee Every journal entry is validated to ensure total debits equal total credits before it can be posted. The system will reject unbalanced entries.

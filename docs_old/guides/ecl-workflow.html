8-Step ECL Workflow

A guided process from data review through to sign-off, with full auditability at every step.

Workflow State Each project tracks its current step (0–7), audit log, overlays, scenario weights, and sign-off status. Steps must be completed in order. The workflow state is stored in Lakebase and persists across sessions.

Step 1: Data & Portfolio Review

Data Processing step — portfolio summary with stage distribution and DQ checks

Purpose

Review the loan portfolio composition, data quality, and stage distribution before proceeding with model execution.

What You See

  • Portfolio Summary — total loans, GCA, and ECL by product type
  • Stage Distribution — count and GCA breakdown across Stage 1, 2, and 3
  • Borrower Segments — distribution by risk segment (Prime, Near-Prime, Sub-Prime)
  • Vintage Analysis — origination vintage cohort breakdown
  • DPD Distribution — days past due histogram
  • PD Distribution — probability of default distribution
  • Data Quality Results — automated DQ checks with pass/fail/warning status

What to Check

  • Are all product types represented?
  • Is the stage distribution reasonable (typically 80–90% Stage 1)?
  • Are there any critical DQ failures?
  • Do the GCA totals reconcile with your source system?

API

POST /api/projects/{project_id}/advance
{
  "action": "data_review",
  "user": "credit.analyst@bank.com",
  "detail": "Portfolio data reviewed and validated",
  "status": "completed"
}

Step 2: Model Execution

Purpose

Run satellite models to calibrate PD and LGD parameters, then review model performance.

What You See

  • Satellite Model Comparison — performance of different algorithms (Linear, Ridge, XGBoost, etc.) across product cohorts
  • Best Model Selection — automatically selected best-performing model per cohort
  • Cohort Summary — model results broken down by product type and risk segment
  • Model Run History — previous model execution runs with metadata

What to Check

  • Are model performance metrics (R², RMSE, MAE) within acceptable ranges?
  • Is the best model selection reasonable for each cohort?
  • Are there any cohorts with poor model fit?

Triggering Model Runs

POST /api/jobs/trigger
{
  "job_key": "satellite_ecl_sync",
  "enabled_models": ["linear_regression", "ridge_regression", "xgboost"]
}

Step 3: ECL Calculation

Purpose

Review the calculated ECL across products, stages, and scenarios. Optionally run Monte Carlo simulations with custom parameters.

What You See

  • ECL by Product — total ECL, GCA, and coverage ratio per product type
  • ECL by Stage — loss allowance breakdown by IFRS 9 stage
  • Scenario Summary — ECL under base, upside, and downside scenarios with weights
  • Monte Carlo Distribution — simulation results showing ECL distribution
  • Top Exposures — largest individual loan exposures

Monte Carlo Simulation

The simulation engine supports configurable parameters:

Step 4: Scenario Analysis

Purpose

Set probability weights for each macroeconomic scenario and review the weighted ECL.

What You See

  • Scenario Weights — adjustable sliders for base, upside, and downside probabilities (must sum to 100%)
  • Weighted ECL Impact — how different weight combinations affect total ECL
  • Scenario Comparison — side-by-side ECL under each scenario

API

POST /api/projects/{project_id}/scenario-weights
{
  "weights": {
    "base": 0.50,
    "upside": 0.20,
    "downside": 0.30
  }
}

Step 5: Management Overlays

Purpose

Apply post-model adjustments (overlays) to account for factors not captured by the models — such as emerging risks, sector-specific concerns, or expert judgment.

What You See

  • Overlay Entry Form — add overlays by product type with amount, type (absolute/percentage), and justification
  • Overlay Summary — total overlay impact on ECL
  • IFRS 9 Reference — link to relevant IFRS 9 paragraphs for each overlay type

API

POST /api/projects/{project_id}/overlays
{
  "overlays": [
    {
      "id": "OVL-001",
      "product": "SME Lending",
      "type": "absolute",
      "amount": 500000,
      "reason": "Emerging sector risk in commercial real estate",
      "ifrs9": "IFRS 9.5.5.17(c)"
    }
  ],
  "comment": "Management overlay for CRE sector risk"
}

Step 6: Stress Testing

Purpose

Review sensitivity analysis and stress test results to understand ECL behavior under adverse conditions.

What You See

  • Sensitivity Analysis — ECL impact of PD/LGD shifts (±10%, ±20%, ±50%)
  • Scenario Comparison — detailed comparison across all scenarios
  • Stress by Stage — how stress scenarios affect each IFRS 9 stage
  • Vintage Performance — ECL trends by origination vintage
  • Concentration Risk — exposure concentration by segment, product, and risk band

Step 7: Reporting & Disclosure

Purpose

Generate regulatory reports and GL journal entries for the ECL calculation.

What You See

  • IFRS 7 Disclosure — comprehensive disclosure package with credit risk exposure, stage migration, and loss allowance reconciliation
  • ECL Movement Schedule — period-over-period ECL movement by driver
  • GL Journals — generated double-entry journal entries ready for posting
  • Stage Migration Matrix — transition matrix with loan counts and GCA

See the GL Journals guide and Regulatory Reports guide for details.

Step 8: Sign-Off

Purpose

Review the complete audit trail and sign off the ECL calculation.

What You See

  • Audit Trail — chronological log of every action taken during the workflow, with timestamps and user identities
  • Summary Dashboard — final ECL figures, scenario weights, overlays, and key metrics
  • Sign-Off Form — enter your name to formally sign off the calculation

API

POST /api/projects/{project_id}/sign-off
{
  "name": "Jane Smith, Chief Risk Officer"
}
Audit Trail Every step advancement, overlay submission, scenario weight change, and sign-off is recorded in the project's audit log with the user identity and timestamp. This provides a complete audit trail for regulatory review.

Resetting a Project

If you need to restart the workflow (e.g., after data corrections), you can reset a project back to Step 0:

POST /api/projects/{project_id}/reset

This clears the step progress but preserves the audit log showing the reset action.

Common Issues

IssueSolution
Cannot advance past Step 1Check that data quality results have no critical errors. Review DQ results at GET /api/data/dq-results.
Scenario weights don't sum to 100%Adjust weights so they sum to exactly 1.0 (100%). The API validates this.
Monte Carlo simulation is slowReduce n_simulations to 500 for faster results. Use the streaming endpoint POST /api/simulate-stream for progress updates.
Overlays not reflected in ECLOverlays are recorded as post-model adjustments. They appear in the attribution waterfall and GL journals but don't modify the underlying ECL calculation.

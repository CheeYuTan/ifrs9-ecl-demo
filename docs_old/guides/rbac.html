RBAC & Governance

Role-based access control, segregation of duties, ECL immutability, and audit trail for regulatory compliance.

Overview

The IFRS 9 ECL Workspace enforces a strict governance framework to ensure that ECL calculations are produced, reviewed, and approved by authorized personnel with a complete audit trail. The system implements four roles with clearly defined permissions, segregation of duties between simulation runners and sign-off authorities, and cryptographic immutability of signed-off ECL results.

Roles

Every user is assigned exactly one of four roles. Roles are stored in the rbac_users table and enforced by the backend on every API call.

RoleDescriptionTypical User
AnalystRuns simulations, reviews data, submits overlays, and generates draft reports. Cannot approve or sign off.Credit Risk Analyst, Model Developer
ReviewerReviews ECL results, backtests, and reports. Can flag issues and add review comments but cannot sign off.Model Validator, Senior Analyst
ApproverSigns off ECL calculations, approves models for production, and finalizes regulatory reports.Chief Risk Officer, Head of Credit Risk
AdminFull system access including user management, configuration, and role assignment. Cannot sign off ECL (segregation of duties).System Administrator, IT Risk

Segregation of Duties

Regulatory frameworks require that the person who produces an ECL calculation cannot also approve it. The platform enforces the following segregation rules:

  • Simulation runner ≠ Sign-off authority — The user who ran the Monte Carlo simulation or advanced the workflow cannot be the same user who signs off the ECL. The API rejects sign-off attempts where the signer matches the simulation runner recorded in the audit log.
  • Model developer ≠ Model validator — The user who registered a model (created_by) cannot validate it. A different user with the Reviewer role must perform validation.
  • Model validator ≠ Model approver — The user who validated a model cannot also approve it for production. An Approver-role user who was not the validator must approve.
  • Admin cannot sign off — Admin users have full system access but are explicitly blocked from signing off ECL calculations to prevent privilege escalation.
Enforcement Segregation of duties is enforced at the API layer, not just the UI. Even direct API calls are validated against the audit trail to ensure no single user can both produce and approve an ECL result.

ECL Immutability After Sign-Off

Once an ECL calculation is signed off, the results become immutable. The platform guarantees integrity through SHA-256 hash verification:

  1. Hash computation — At sign-off, the system computes a SHA-256 hash over the complete ECL result set (scenario weights, weighted ECL by product and stage, overlays, and simulation parameters).
  2. Hash storage — The hash is stored alongside the sign-off record in the ecl_projects table, linked to the signer identity and timestamp.
  3. Tamper detection — Any subsequent read of a signed-off project recomputes the hash and compares it to the stored value. A mismatch triggers an integrity alert.
  4. No modification — The API rejects any attempt to modify overlays, scenario weights, or advance the workflow on a signed-off project. The only permitted action is to create a new project or reset (which preserves the original audit trail).
GET /api/projects/{project_id}

{
  "project_id": "Q4-2025-ECL",
  "signed_off_by": "Jane Smith, CRO",
  "signed_off_at": "2025-12-31T14:30:00Z",
  "ecl_hash": "a3f2b8c1d4e5...sha256...",
  "integrity_check": "PASS"
}

Audit Trail

Every action in the platform is recorded in the project audit log with the authenticated user identity. The audit trail captures:

FieldDescription
actionThe action performed (e.g., data_review, simulation_run, overlay_submitted, sign_off)
userAuthenticated user identity (email or display name)
timestampISO 8601 timestamp of the action
detailAdditional context (e.g., simulation parameters, overlay amounts, review comments)
ip_addressClient IP address for traceability
roleThe user's role at the time of the action

Retrieve the audit trail for any project:

GET /api/projects/{project_id}

// The audit_log field contains the full chronological trail:
{
  "audit_log": [
    {
      "action": "data_review",
      "user": "analyst@bank.com",
      "role": "analyst",
      "timestamp": "2025-12-15T09:00:00Z",
      "detail": "Portfolio data reviewed — 84,000 loans across 5 products"
    },
    {
      "action": "simulation_run",
      "user": "analyst@bank.com",
      "role": "analyst",
      "timestamp": "2025-12-15T10:30:00Z",
      "detail": "Monte Carlo: 1000 iterations, PD-LGD correlation 0.30"
    },
    {
      "action": "sign_off",
      "user": "cro@bank.com",
      "role": "approver",
      "timestamp": "2025-12-31T14:30:00Z",
      "detail": "ECL signed off — total weighted ECL: $42.3M"
    }
  ]
}
Immutable Audit Log The audit log is append-only. Even a project reset preserves the existing audit entries and adds a new "reset" entry. Audit records cannot be deleted or modified through any API endpoint.

Setting Up Users and Assigning Roles

Users are managed through the Admin Configuration panel or via the RBAC API. Only users with the Admin role can create or modify user accounts.

Listing Users

GET /api/rbac/users
GET /api/rbac/users?role=approver

Creating a User

Users are seeded during the setup wizard or created manually by an admin. The seed-defaults endpoint creates a default set of users:

POST /api/admin/seed-defaults

This creates default users for each role:

Approval Workflows

Critical actions (model approval, ECL sign-off, report finalization) require formal approval requests that route to the appropriate authority:

POST /api/rbac/approvals
{
  "request_type": "ecl_signoff",
  "entity_id": "Q4-2025-ECL",
  "entity_type": "project",
  "requested_by": "analyst-001",
  "assigned_to": "approver-001",
  "priority": "high",
  "due_date": "2025-12-31",
  "comments": "Q4 ECL ready for CRO sign-off"
}

The approver can then approve or reject:

POST /api/rbac/approvals/{request_id}/approve
{
  "user_id": "approver-001",
  "comment": "Reviewed and approved"
}
Regulatory Compliance The combination of role-based access, segregation of duties, SHA-256 immutability, and append-only audit trails satisfies the governance requirements of IFRS 9, Basel III/IV, and most national banking regulators for ECL calculation processes.

Domain Validation Rules

23-rule validation engine ensuring data integrity, model reasonableness, and governance compliance at every stage of the ECL workflow.

Overview

The validation engine runs automatically at key points in the ECL workflow to catch data errors, model anomalies, and governance violations before they propagate into final ECL figures. Rules are organized into three categories:

CategoryRule CountPurpose
Data Integrity (D)6 rules (D-1 to D-6)Ensure input data is complete, consistent, and within valid ranges
Model Reasonableness (M-R)8 rules (M-R1 to M-R8)Verify that model outputs and ECL results are economically sensible
Governance (G-R)5 rules (G-R1 to G-R5)Enforce process controls, approvals, and regulatory requirements

Additionally, 4 cross-cutting rules (X-1 to X-4) validate consistency across categories, bringing the total to 23 rules.

Severity Levels

SeverityEffectDescription
CriticalBlocks workflowECL calculation cannot proceed. Must be resolved before advancing.
WarningFlags for reviewECL calculation can proceed, but the issue is flagged for reviewer attention before sign-off.
InfoLogged onlyInformational finding recorded in the audit trail. No action required.

Data Integrity Rules (D-1 to D-6)

These rules run during Step 1: Data & Portfolio Review as pre-calculation gates.

Rule IDDescriptionSeverityEnforcement Point
D-1Required fields not null — loan_id, product_type, assessed_stage, gross_carrying_amount must not be NULL.CriticalPre-calculation
D-2Stage values valid — assessed_stage must be 1, 2, or 3. No other values permitted.CriticalPre-calculation
D-3PD within bounds — current_lifetime_pd and current_12m_pd must be between 0.0 and 1.0 (inclusive).CriticalPre-calculation
D-4LGD within bounds — lgd must be between 0.0 and 1.0 (inclusive).CriticalPre-calculation
D-5GCA positive — gross_carrying_amount must be greater than zero.CriticalPre-calculation
D-6No duplicate loan IDs — Each loan_id must appear exactly once in the dataset.CriticalPre-calculation

Model Reasonableness Rules (M-R1 to M-R8)

These rules run during Step 3: ECL Calculation and Step 6: Stress Testing as post-calculation checks.

Rule IDDescriptionSeverityEnforcement Point
M-R1ECL ≤ GCA — No loan's ECL amount can exceed its gross carrying amount. Coverage ratio must be ≤ 100%.CriticalPost-calculation
M-R2Stage monotonicity — Average coverage ratio must increase from Stage 1 → Stage 2 → Stage 3 (i.e., higher stages = higher loss rates).WarningPost-calculation
M-R3DPD-stage consistency — Stage 1 loans should have DPD < 30; Stage 3 loans should have DPD ≥ 90. Exceptions flagged.WarningPre-calculation
M-R4Scenario ordering — Downside ECL ≥ Base ECL ≥ Upside ECL at the portfolio level.WarningPost-calculation
M-R5Scenario weights sum to 1 — Base + upside + downside weights must equal exactly 1.0 (100%).CriticalPre-calculation
M-R6Coverage ratio bounds — Portfolio-level coverage ratio should be between 0.5% and 25%. Values outside this range suggest data or model issues.WarningPost-calculation
M-R7Period-over-period ECL movement — Total ECL change exceeding ±50% from the prior period triggers a review flag.WarningPost-calculation
M-R8Concentration limit — No single product type should represent more than 80% of total ECL unless the portfolio is genuinely single-product.InfoPost-calculation

Governance Rules (G-R1 to G-R5)

These rules run during Step 7: Reporting and Step 8: Sign-Off to enforce process controls.

Rule IDDescriptionSeverityEnforcement Point
G-R1All steps completed — Sign-off requires all 7 preceding workflow steps to be marked as completed.CriticalSign-off gate
G-R2Segregation of duties — The sign-off user must differ from the user who ran the simulation (see RBAC & Governance).CriticalSign-off gate
G-R3Champion model required — At least one champion model must exist in the model registry for the ECL calculation to be signed off.WarningSign-off gate
G-R4Backtest currency — The most recent backtest for the champion PD model must be less than 90 days old.WarningSign-off gate
G-R5Overlay justification — Every management overlay must have a non-empty reason field and an IFRS 9 paragraph reference.CriticalOverlay submission

Cross-Cutting Rules (X-1 to X-4)

Rule IDDescriptionSeverityEnforcement Point
X-1GL reconciliation — Total ECL from the calculation engine must match total journal entry amounts within a 0.01 tolerance.CriticalPost-journal generation
X-2Double-entry balance — Total debits must equal total credits for every journal entry.CriticalJournal posting
X-3Vintage format — vintage_cohort values should follow the YYYY-QN format (e.g., "2023-Q1").InfoPre-calculation
X-4Reporting date consistency — All loans in the dataset should share the same reporting_date.WarningPre-calculation

Pre-Calculation Gates

Before the ECL calculation can proceed past Step 1, all Critical pre-calculation rules must pass. If any critical rule fails, the workflow is blocked:

POST /api/projects/{project_id}/advance
{
  "action": "data_review",
  "user": "analyst@bank.com",
  "status": "completed"
}

// Response when critical rules fail:
{
  "error": "Cannot advance: 2 critical validation failures",
  "failures": [
    {"rule": "D-1", "message": "142 loans have NULL product_type"},
    {"rule": "D-3", "message": "8 loans have PD > 1.0"}
  ]
}

The analyst must fix the underlying data issues and re-run the data quality checks before the workflow can advance.

Post-Calculation Checks

After ECL calculation (Step 3) and before sign-off (Step 8), post-calculation rules are evaluated and their results are attached to the project. Warning-level findings do not block the workflow but are surfaced prominently in the sign-off review screen:

GET /api/data/dq-results

[
  {"rule": "M-R2", "severity": "warning", "status": "fail",
   "message": "Stage 2 avg coverage (1.8%) is lower than Stage 1 (2.1%)"},
  {"rule": "M-R4", "severity": "warning", "status": "pass",
   "message": "Scenario ordering correct: downside > base > upside"},
  {"rule": "M-R6", "severity": "warning", "status": "pass",
   "message": "Portfolio coverage ratio 3.2% within expected range"}
]
Reviewer Responsibility Warning-level findings must be reviewed and acknowledged by a Reviewer or Approver before sign-off. The audit trail records which warnings were present at the time of sign-off, ensuring regulatory transparency.

Validation Dashboard

The ECL Workflow UI includes a validation dashboard that provides a real-time view of all 23 rules:

  • Summary bar — Shows pass/fail/warning counts at a glance
  • Category breakdown — Data Integrity, Model Reasonableness, and Governance sections with expandable detail
  • Trend view — How validation results have changed across recent ECL runs
  • Drill-down — Click any failed rule to see the specific loans or values that triggered the failure

Access validation results programmatically:

GET /api/data/dq-results      // Detailed rule-by-rule results
GET /api/data/dq-summary       // Aggregated pass/fail/warning counts
Extensibility The validation engine is designed to be extensible. Custom rules can be added by defining new rule functions in the backend validation module. Each rule specifies its ID, description, severity, enforcement point, and validation logic.

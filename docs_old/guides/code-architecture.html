Code Architecture

Modular backend structure organized by domain responsibility. Each package owns its data access, business logic, and API routes, with strict separation of concerns and a 200-line file size limit.

Package Responsibilities

domain/ — Business Logic

The domain/ package contains all business logic as pure functions with no HTTP or framework dependencies. Each module handles a specific domain concept:

Module GroupFilesResponsibility
ECL Core queries.py, model_runs.py, workflow.py Loan data access, ECL run orchestration, and the 8-step workflow state machine that governs project progression from data review through sign-off.
Validation validation_rules.py 29 validation rules across four categories: Data Integrity (D-1 to D-10), Model Reasonableness (M-R1 to M-R8), Governance (G-R1 to G-R5), and Domain Accuracy (DA-1 to DA-6). See Validation Rules and Domain Accuracy Rules.
Models model_registry.py, data_mapper.py Model version control with champion/challenger tracking, and data mapping between source schemas and the ECL calculation schema.
Analytics attribution.py, markov.py, advanced.py ECL waterfall attribution, Markov chain transition matrices for stage migration, and advanced features (CCF estimation, collateral haircuts, cure rates).
Backtesting backtesting.py, backtesting_stats.py, backtesting_traffic.py Backtest execution, statistical metrics (AUC, Gini, KS, PSI, Brier score), and traffic-light framework for model performance monitoring.
Hazard Models hazard.py, hazard_tables.py, hazard_retrieval.py, hazard_analysis.py, hazard_cox_ph.py, hazard_nonparam.py, hazard_estimation.py Survival analysis suite: Cox Proportional Hazards, Kaplan-Meier estimator, discrete-time hazard models, and survival curve analysis. The hazard.py module serves as a facade that delegates to specialized sub-modules.
Governance audit_trail.py, period_close.py, health.py Immutable audit log, period close workflow with lock/unlock, and system health checks.

routes/ — HTTP Controllers

The routes/ package contains thin FastAPI routers. Each file registers endpoints for a single domain area. Route handlers perform request validation, call into domain/ functions, and return responses. They contain no business logic.

Design Principle Routes are thin wrappers. If you need to add business logic, it belongs in domain/. If you need to add a new endpoint, it belongs in routes/. A route file should never exceed 200 lines.

Hazard Models — Sub-Module Architecture

The hazard model subsystem in domain/ follows a facade pattern. The top-level hazard.py provides a unified API while delegating to specialized implementations:

domain/
  hazard.py              # Facade: public API, model selection
    hazard_tables.py     # Storage: save/load survival tables
    hazard_retrieval.py  # Query: fetch fitted models and curves
    hazard_analysis.py   # Analysis: survival curve comparison, CI bands
    hazard_cox_ph.py     # Cox PH: semi-parametric proportional hazards
    hazard_nonparam.py   # Non-parametric: Kaplan-Meier, discrete-time
    hazard_estimation.py # Utilities: hazard rate and cumulative hazard

This decomposition keeps each file under 200 lines while maintaining a clean public interface. External code should only import from hazard.py; the sub-modules are internal implementation details.

Design Principles

  • 200-line file limit — No Python file exceeds 200 lines of logic. When a module grows beyond this, it is decomposed (as done with hazard models and IFRS 7 reports).
  • Separation of concerns — routes/ handles HTTP, domain/ handles logic, reporting/ handles output generation, db/ handles connections, middleware/ handles cross-cutting.
  • Facade pattern — Complex subsystems (hazard models, reporting) expose a single public module that delegates to internal sub-modules.
  • No circular imports — Dependencies flow one way: routes imports domain; domain imports db; nothing imports routes.
  • Testability — Business logic in domain/ is pure functions that can be unit-tested without HTTP or database dependencies.

Adding a New Feature To add a new feature: (1) create a domain module in domain/ with business logic, (2) create a route file in routes/ with API endpoints, (3) register the router in app.py, (4) add a frontend page in frontend/src/pages/. If the feature needs reports, add a sub-module in reporting/.

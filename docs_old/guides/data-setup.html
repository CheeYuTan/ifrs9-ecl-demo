Data Setup

Prepare and map your loan portfolio data for ECL calculation.

Required Input Tables

The application reads from five Unity Catalog tables. These are populated by the data processing and ECL calculation pipelines (scripts 01_generate_data.py through 04_sync_to_lakebase.py).

model_ready_loans

The primary loan-level input table containing all attributes needed for ECL calculation.

ColumnTypeDescription
loan_idSTRINGUnique loan identifier
product_typeSTRINGProduct category (e.g., Mortgages, Personal Loans, Auto Finance, Credit Cards, SME Lending)
segmentSTRINGBorrower segment (e.g., Prime, Near-Prime, Sub-Prime)
assessed_stageINTIFRS 9 stage assignment: 1 (performing), 2 (significant increase in credit risk), 3 (credit-impaired)
original_stageINTStage at the beginning of the reporting period (for migration analysis)
days_past_dueINTNumber of days the loan is past due
gross_carrying_amountDOUBLEGross carrying amount (GCA) of the loan
current_lifetime_pdDOUBLECurrent lifetime probability of default (0.0 – 1.0)
current_12m_pdDOUBLE12-month probability of default (0.0 – 1.0)
lgdDOUBLELoss Given Default (0.0 – 1.0)
eadDOUBLEExposure at Default
ecl_amountDOUBLECalculated ECL amount for this loan
remaining_monthsINTRemaining term to maturity in months
vintage_cohortSTRINGOrigination vintage (e.g., "2023-Q1")
risk_bandSTRINGRisk rating band (e.g., "A", "B", "C", "D")
collateral_typeSTRINGType of collateral (e.g., "Property", "Vehicle", "None")
collateral_valueDOUBLECurrent collateral valuation
interest_rateDOUBLEContractual interest rate
origination_dateDATELoan origination date
maturity_dateDATELoan maturity date
reporting_dateDATEReporting period end date

loan_ecl_weighted

Scenario-weighted ECL at the loan level — the final ECL after applying scenario weights.

ColumnTypeDescription
loan_idSTRINGUnique loan identifier
product_typeSTRINGProduct category
assessed_stageINTIFRS 9 stage (1, 2, or 3)
gross_carrying_amountDOUBLEGross carrying amount
weighted_eclDOUBLEProbability-weighted ECL across all scenarios
base_eclDOUBLEECL under the base scenario
upside_eclDOUBLEECL under the upside scenario
downside_eclDOUBLEECL under the downside scenario
coverage_ratioDOUBLEECL / GCA ratio

loan_level_ecl

Per-scenario ECL at the loan level — one row per loan per scenario.

ColumnTypeDescription
loan_idSTRINGUnique loan identifier
scenarioSTRINGScenario name (base, upside, downside)
product_typeSTRINGProduct category
assessed_stageINTIFRS 9 stage
pdDOUBLEScenario-adjusted probability of default
lgdDOUBLEScenario-adjusted loss given default
eadDOUBLEExposure at default
eclDOUBLECalculated ECL for this scenario
gross_carrying_amountDOUBLEGross carrying amount

portfolio_ecl_summary

Aggregated portfolio-level ECL by product type, stage, and scenario.

ColumnTypeDescription
product_typeSTRINGProduct category
assessed_stageINTIFRS 9 stage
scenarioSTRINGScenario name
loan_countINTNumber of loans in this segment
total_gcaDOUBLETotal gross carrying amount
total_eclDOUBLETotal ECL amount
avg_pdDOUBLEAverage probability of default
avg_lgdDOUBLEAverage loss given default
coverage_pctDOUBLEECL coverage ratio (ECL / GCA × 100)
weightDOUBLEScenario probability weight

ifrs7_stage_migration

Period-over-period stage transition counts for IFRS 7 disclosure.

ColumnTypeDescription
original_stageINTStage at beginning of period
assessed_stageINTStage at end of period
product_typeSTRINGProduct category
loan_countINTNumber of loans that migrated
total_gcaDOUBLETotal GCA of migrated loans
total_eclDOUBLETotal ECL of migrated loans

Column Mapping

If your source data uses different column names, the Admin Configuration panel provides a mapping interface. Navigate to Admin → Data Sources to:

  1. Select your Unity Catalog table
  2. The system auto-suggests mappings based on column name similarity
  3. Review and adjust mappings manually
  4. Validate that all required columns are mapped

You can also use the API:

# Get auto-suggested mappings
curl https://<app-url>/api/admin/suggest-mappings/model_ready_loans

# Validate a mapping
curl -X POST https://<app-url>/api/admin/validate-mapping \
  -H "Content-Type: application/json" \
  -d '{
    "table_key": "model_ready_loans",
    "mappings": {
      "loan_id": "account_number",
      "product_type": "product_category",
      "assessed_stage": "ifrs9_stage"
    }
  }'

Data Quality Requirements

The application runs data quality checks during Step 1 of the workflow. Key requirements:

CheckRequirementSeverity
Completenessloan_id, product_type, assessed_stage must not be NULLError
Stage Valuesassessed_stage must be 1, 2, or 3Error
PD Rangecurrent_lifetime_pd must be between 0 and 1Error
LGD Rangelgd must be between 0 and 1Error
GCA Positivegross_carrying_amount must be > 0Error
DPD ConsistencyStage 1 loans should have DPD < 30; Stage 3 should have DPD ≥ 90Warning
Duplicate LoansNo duplicate loan_id valuesError
Vintage Formatvintage_cohort should follow YYYY-QN formatWarning
